# Knowledge files and `.know` directory structure

Know stores its project data in a `.know/` directory at the repository root.

## Terminology

Use these terms consistently across documentation:

- **Knowledge files**: the human-authored TOML files under `.know/rules/` and `.know/concepts/`. This is the everyday term. "Open the billing knowledge file." "Write your rules in knowledge files."
- **TOML files**: the technical precision term for the same files, used when the format itself is relevant. "Knowledge files are plain TOML committed to Git." Individual files can be named by content and type: "billing rules TOML file", "customer-identity concepts TOML file".
- **Generated files**: `.know/linkVerification.lock.toml` and `.know/cache/` content — produced by Know, not authored by hand.

## Structure

The `.know/` directory has source definitions, verification files, and disposable generated cache data:

```txt
.know/
├── rules/
│ ├── labels.toml
│ ├── billing.toml
│ └── ...
├── concepts/
│   ├── customer-identity.toml
│   ├── billing.toml
│   └── ...
├── linkVerification.toml
├── linkVerification.lock.toml
├── .gitignore
└── cache/
    ├── know.sqlite
    └── semantic/
```

Links are defined inline inside rule definitions. A link is the relationship between one rule and one code target.

## File behavior

- **Knowledge files** (TOML files under `.know/rules/` and `.know/concepts/`) are human-authored and committed to Git.
- **`.know/linkVerification.toml`** is generated by Know and stores the last verified fingerprints for rule-link-code relationships.
- **`.know/linkVerification.lock.toml`** is generated by Know and stores the last computed link-verification reflection for the current repository.
- **`.know/cache/`** is disposable local generated data (SQLite read model and semantic index).

`know init` creates `.know/.gitignore` so `cache/` is ignored by Git, while knowledge files and verification files remain committable.

## Source of truth and generation flow

Know parses current knowledge files, reads `.know/linkVerification.toml`, resolves linked targets in the repository, and writes derived status reflection to `.know/linkVerification.lock.toml`. SQLite is generated from the same analysis for fast command and search workflows.

The source of truth is the current knowledge files plus `.know/linkVerification.toml`. The lockfile is a committed generated reflection. Cache content under `.know/cache/` is disposable.
