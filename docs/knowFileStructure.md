# Know files and file structure

The Know files are stored in a .know directory in the root of the project. This directory contains all rule and concept definitions that are relevant to the project.

Definitions are documented in .toml files.

Each directory has one or many .toml definition files.
File names are arbitrary and ignored by the system.
Files in each directory are parsed for definitions that are added to the Know system.

Links are defined inline inside rule definitions. A link is the relationship between one rule and one code target.

The `.know` directory has one sub-directory for each source-defined primitive, a generated link-verification approval file, a generated link-verification lockfile, and an ignored cache directory for disposable local indexes:

```txt
.know/
├── rules/
│ ├── labels.toml
│ ├── billing.toml
│ └── ...
├── concepts/
│   ├── customer-identity.toml
│   ├── billing.toml
│   └── ...
├── linkVerification.toml
├── linkVerification.lock.toml
├── .gitignore
└── cache/
    ├── know.sqlite
    └── semantic/
```

The file structure is designed for ease of use, readability, and discoverability. Users and AI-agents can easily navigate the directory structure to find and edit the relevant definitions. The use of .toml files allows for a structured and human-readable format for defining rules, concepts, and inline rule links.

The source-defined rule and concept files are human-authored. `.know/linkVerification.toml` is generated by Know and stores the last verified fingerprints for rule-link-code relationships. Know writes it deterministically with normalized values and stable ordering. It records user-approved fingerprints; it is not regenerated from current fingerprints unless the user verifies the relationship.

`.know/linkVerification.lock.toml` is generated by Know and stores the last computed link-verification reflection for the current repository. It is committed to Git so derived status changes are reviewable. It is not authoritative unless `know check` proves it is fresh against the current `.know` source files and repository code.

`.know/cache/` contains generated local data, including the SQLite read model and semantic search index. It is disposable and can be deleted and rebuilt by `know index`. `know init` creates `.know/.gitignore` so `cache/` is ignored by Git while source TOML files, `linkVerification.toml`, and `linkVerification.lock.toml` remain committable.

The know system parses the `.know` source files, reads `.know/linkVerification.toml`, resolves linked targets in the current repository, and writes the derived link-verification reflection to `.know/linkVerification.lock.toml`. SQLite is a generated read model built from the same current analysis for fast command and search workflows. The original `.know` source files and `.know/linkVerification.toml` remain the approval source of truth; `.know/linkVerification.lock.toml` is the committed generated reflection; `.know/cache/` is disposable.
